# Session Log — 2026-07-21

## Session goal

Turn HERO Problem Solve defect information from a generic event label into stable, operationally useful evidence, then eliminate visible startup/render delay in POPS and HERO.

## Starting point

HERO Problem Solve card could identify that a defect-related event occurred, but several fields were incomplete or unstable.

Observed card state included:

- Associate: blank
- Defect: blank or generic
- Defect Time: populated
- Event: `ReportDefect`
- PS Tote: blank
- Previous Condition: blank
- Source: blank
- Module / Team: partly populated

The page sometimes showed richer information briefly and then lost it.

## Investigation 1 — Why values disappeared

### Observation

Martins reported that the card initially showed a real reason such as damaged, but then something refreshed and erased it across several rows.

### Diagnosis

The main HERO module reset Problem Solve fields on every refresh:

- `.associate`
- `.reason`
- `.time`

A separate Core component read richer failure evidence asynchronously. Bootstrap/module startup also triggered overlapping refreshes. The richer reader could finish first, then a slower original HERO refresh could reset its values.

This was a race and ownership problem, not an API failure.

### Fix: HERO Failure Intelligence `0.2.0`

Implemented:

- complete evidence object built before writing;
- exact route binding using URL, FC, and shipment;
- field overwrite MutationObserver;
- automatic evidence repair after another script clears values;
- request generation invalidation on route change;
- extension fields cleared safely between shipments;
- deduplicated retry timers.

### Result

Martins confirmed the evidence remained visible.

## Investigation 2 — Why `ReportDefect` was still not enough

### Observation

The card now remained stable, but the Defect value still effectively said only that the item was reported defective.

Martins correctly identified the required operational categories:

- item missing;
- transparency / serial-number issue;
- damaged;
- ideally unscannable as another useful category.

### Important semantic distinction

`ReportDefect` is the event.

It is not the reason.

The card needs both:

- Event: `ReportDefect`
- Defect: `Damaged`, `Item missing`, or `Transparency / serial number issue`

### Evidence already confirmed

Previous Scout investigation confirmed HERO event-details can contain specific nested evidence such as:

- item-level `failureReason`;
- `DefectReasonCode`;
- `DefectReasonText`;
- `shipmentFailureReason`;
- `DefectType`;
- Problem Solve tote/container;
- previous external shipment condition;
- shipment group/source;
- client application/module information.

A confirmed example contained item-level:

`ITEM_TRANSPARENCY_VALIDATION_FAILURE`

while broader shipment-level evidence was null or less useful.

### Fix: HERO Failure Intelligence `0.3.0`

Parser priority became:

1. item-level `failureReason`;
2. `DefectReasonCode`;
3. `DefectReasonText`;
4. `shipmentFailureReason`;
5. `DefectType`;
6. generic event description.

Classification mapping added:

- transparency/serial indicators → `Transparency / serial number issue`;
- missing indicators → `Item missing`;
- damage indicators → `Damaged`;
- barcode/unscannable indicators → `Unscannable`;
- no specific evidence → `Defect reason not exposed`.

Added a separate `Defect Evidence` row containing the raw code used for classification.

The reader now inspects up to eight newest relevant `ReportDefect` and `FAIL_SHIPMENT` events and skips a generic newest event when an older specific failure event exposes the real reason.

### Release

- HERO Failure Intelligence: `0.3.0`
- Manifest: `2.6.38`

### Result

Martins confirmed it worked and highlighted that the feature was running through Nova Core rather than a standalone script.

## Investigation 3 — Visible page delay / flicker

### Observation

Martins noticed two timing flaws:

### POPS

On refresh:

1. original POPS UI appeared;
2. less than a second later Nova modern styling replaced it.

### HERO

On opening a shipment:

1. the Nova panel appeared;
2. data changed suddenly as refresh/API work completed.

Martins expected this might be an unavoidable framework trade-off.

### Diagnosis

It was not an unavoidable trade-off.

Bootstrap `2.2.5` waited until `DOMContentLoaded` before starting Nova and resolved the live GitHub manifest before fully loading Core. Even when code was cached, the live update path was still in the critical startup chain.

POPS therefore painted before its module ran.

HERO revealed its panel before the event list and all event-detail requests had settled.

## Fix: Bootstrap Fast Boot `2.3.0`

New normal refresh order:

1. read cached active manifest immediately;
2. execute cached Core components;
3. launch cached matching modules;
4. check GitHub later in the background;
5. stage updates for the next refresh.

First install still downloads required files because no cache exists.

Bootstrap continues to run at `document-start`.

## Fix: Nova Site Render Gate `1.0.0`

New Core component:

`core/nova-site-render-gate.js`

### POPS readiness

POPS is revealed when its module reports:

- active;
- visible;
- modern styling applied to operational controls/cards.

### HERO readiness

HERO is revealed when:

- the Nova panel exists and is visible;
- panel loading state is removed;
- the main shipment-event request has occurred;
- tracked shipment event/detail request count returns to zero;
- a quiet window confirms no late request is still changing the panel.

### Failure protection

A maximum timeout releases the page even when readiness detection fails, avoiding a permanent loading cover.

### Release

- Bootstrap: `2.3.0`
- Nova Site Render Gate: `1.0.0`
- Manifest: `2.6.39`
- Manifest minimum Bootstrap: `2.3.0`

Because Bootstrap itself changed, Tampermonkey must update the installed Bootstrap before the new startup path can work.

## Current architectural follow-up

HERO Failure Intelligence is still a separate Core compatibility layer writing some fields also owned by the main HERO module.

Recommended cleanup after the current behaviour is proven:

1. move failure parsing/classification into `modules/hero/nova-hero-intelligence.js`;
2. add one request-generation controller for the main refresh;
3. await all required event-detail requests before marking the panel complete;
4. make one component the only writer of Problem Solve fields;
5. remove or reduce the Core failure component to a compatibility wrapper.

Do not perform this cleanup before validating the current working behaviour, because the separate component currently protects production usability.

## Key lesson from this session

Two phrases drove the important improvements:

> “It does show, but we are still missing crucial information.”

and:

> “The delay… you can see the old original page.”

Both were examples of a feature being technically functional but below the real quality bar.

The first produced better operational semantics.

The second produced a faster and more controlled rendering architecture.
